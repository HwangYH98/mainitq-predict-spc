# 5월 11일 발표용 3분 Streamlit 리허설 대본

## 실행 준비

- 실행 파일: `run_dashboard.bat`
- 직접 실행 명령어: `.\.venv\Scripts\python.exe -m streamlit run app\dashboard.py --server.headless true --browser.gatherUsageStats false`
- 접속 주소: `http://127.0.0.1:8501`
- 발표 전 브라우저 창은 가로로 넓게 열어 주요 탭이 한 줄에 보이도록 준비합니다.
- 현재 구현 상태는 **Stage 1~6-lite 완료**, **Stage 7~8-lite 발표용 MVP 구현**으로 말합니다.

## 3분 발표 흐름

### 0:00-0:25 | 성과 요약 탭

클릭: `성과 요약`

발표:
"안녕하세요. 제 캡스톤은 제조 설비 센서 데이터를 이용해 기계 고장 위험을 예측하고, 그 결과와 판단 근거를 Streamlit 대시보드로 보여주는 predictive maintenance PoC입니다. 현재 Stage 1부터 6-lite까지는 완료했고, Stage 7과 8-lite는 발표용 MVP 수준으로 구현했습니다. 핵심 결과는 XGBoost가 PR-AUC 기준 대표 모델이고, threshold를 0.87로 조정했을 때 F1-score가 0.7752까지 개선되었다는 점입니다."

### 0:25-0:55 | 모델 비교 탭

클릭: `모델 비교`

발표:
"먼저 baseline 모델 비교입니다. Logistic Regression은 가장 기본적인 기준 모델이고, XGBoost는 센서 변수 사이의 비선형 관계를 더 잘 잡기 위한 비교 모델입니다. test set은 2,000개 row이고 실제 고장 row는 68개라 클래스 불균형이 있습니다. 그래서 accuracy만 보지 않고 precision, recall, F1-score, ROC-AUC, PR-AUC를 함께 봤습니다. 그 결과 XGBoost의 PR-AUC는 0.8014로 Logistic Regression의 0.3817보다 높았습니다."

### 0:55-1:25 | Threshold 조정 탭

클릭: `Threshold 조정`

발표:
"다음은 threshold 조정입니다. 모델은 고장 확률을 출력하고, threshold는 그 확률을 High Risk로 볼 기준입니다. 기본 threshold 0.50에서는 recall은 높지만 precision이 낮아 false alarm이 많아질 수 있습니다. 이번 결과에서는 F1-score 기준으로 threshold 0.87을 선택했고, F1-score가 0.5911에서 0.7752로 좋아졌습니다. 이 부분은 현장에서 경고 기준을 어떻게 정할지와 연결됩니다."

### 1:25-1:50 | SHAP 해석 탭

클릭: `SHAP 해석`

발표:
"이제 모델이 왜 고장이라고 판단했는지 설명하는 부분입니다. SHAP은 XGBoost 예측에 어떤 센서 변수가 영향을 줬는지 보여줍니다. 여기서는 torque, rotational speed, tool wear 같은 값이 주요 근거로 나타났습니다. 그래서 단순히 고장 확률만 보여주는 것이 아니라, 어떤 센서 상태 때문에 위험하다고 판단했는지 설명할 수 있습니다."

### 1:50-2:15 | 개별 사례 탭

클릭: `개별 사례`

발표:
"이 탭은 실제 고장 사례 하나를 자세히 보여줍니다. UDI 6498, Product ID L53677인 test row는 실제 고장이고, XGBoost 고장 확률이 0.9936입니다. 선택 threshold 0.87을 넘기 때문에 High Risk로 판단됩니다. SHAP 기준으로 torque 65.3, rotational speed 1312, tool wear 192 같은 값이 고장 방향으로 예측을 밀어 올린 주요 근거입니다."

### 2:15-2:40 | Row 시뮬레이션 탭

클릭: `Row 시뮬레이션`

발표:
"이 화면은 실제 센서 스트리밍은 아니고, 저장된 test set 예측 결과를 한 row씩 넘겨보는 playback입니다. 슬라이더를 움직이면 해당 row의 XGBoost 고장 확률과 threshold 0.87 기준의 상태가 Normal 또는 High Risk로 바뀝니다. 고장 사례를 직접 보여주려면 `Test row 선택`을 1296 근처로 이동하면, 방금 본 UDI 6498 사례가 High Risk로 표시됩니다."

### 2:40-2:55 | 발표 요약 탭

클릭: `발표 요약`

발표:
"정리하면, 현재까지는 데이터 준비, baseline 모델링, threshold 조정, SHAP 해석, 대시보드 시각화, row playback까지 구현했습니다. 여기에 CSV 업로드 예측 MVP와 SHAP 기반 처방 초안까지 붙여서, 실제 현장 데이터로 확장할 수 있는 구조를 보여주는 것이 이번 중간발표의 핵심입니다."

### 2:55-3:00 | 마무리

발표:
"다만 아직 실시간 센서 연동, 실제 LLM 호출, 클라우드 배포 단계는 아닙니다. 현재는 로컬 PoC이고, 다음 단계에서 실제 현장 데이터 재검증과 통합 대시보드로 확장할 계획입니다."
